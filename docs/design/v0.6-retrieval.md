# v0.6 — Retrieval Index & Focused Briefs

## Status

Design for implementation. Concretizes the retrieval pillar of
[roadmap.md](roadmap.md) (hardened across three review rounds) into a
buildable increment. Ships as v0.6.

## Problem

Every `brief`/`query` re-reads and re-parses every Markdown file, then returns
*all* active records. Correct and drift-free at small scale, but two things
break as a vault fills (team × months, especially under auto mode): reads get
slow, and an all-records brief becomes a wall nobody reads. Unread memory is
dead memory.

## Design

### 1. Revision-safe derived index

A rebuildable index per vault at
`${XDG_CONFIG_HOME}/context-vault/index/<vault-key>.json`, directory `0700`
(it holds full record text — same sensitivity as the ledger).

- **Correctness key.** For git vaults: current `HEAD` sha plus a digest of
  tracked record files. For non-git vaults: a digest of `(path, size,
  mtime_ns)` for every record, verified by content hash before reuse. mtimes
  alone are never trusted (they lie across git checkout/fast-forward).
- **Contents.** Per record: id, type, project, `recorded_at`, `repos`,
  `consent`, the searchable fields (subject/relation/value, title/choice/
  rationale, completed/next_step), a tokenized term→count map, and the source
  path. Withdrawal and supersession relationships are recorded so ranking can
  respect current-state visibility.
- **Lifecycle.** `load_or_build_index(vault)` returns the index if its
  revision key matches, else rebuilds atomically (temp file + replace) under
  the existing per-vault lock. If the revision can't be established, callers
  fall back to a direct scan and say so (`index: "unavailable"`).
- **Canonical data stays the Markdown.** The index is disposable; deleting it
  forces a clean rebuild. It never holds a fact the records don't.

### 2. Deterministic ranking (documented, tunable)

Not tuned BM25 — the volume to tune it doesn't exist yet. A transparent
scorer, specified so golden-fixture tests make every change intentional:

- **Tokenization.** Lowercase; split on non-alphanumeric; `[[target|alias]]`
  and `[[target]]` both contribute `target`'s tokens plus the literal link;
  a small stopword set dropped.
- **Score.** For a `--focus` query, sum over query tokens of
  `field_weight × term_frequency_in_record`, where field weights are
  subject/title/relation > value/choice > rationale/completed/next_step >
  evidence. Add a recency bonus (bounded decay on `recorded_at`) and a type
  weight (decisions and disputes rank above facts above sessions, all else
  equal). Ties break on `recorded_at` then id, so ordering is stable.
- **Non-droppable safety classes** are always included regardless of score:
  active disputes, active decisions, repair chores, and sync/staleness
  warnings. Ranking chooses what *else* fills the budget.

### 3. Focused briefs

`brief --focus "<what I'm about to do>"` returns the ranked top-N per section
(default caps: 15 facts, 8 decisions, 8 sessions) instead of everything.
Omission is honest: `omitted: {facts: N, ...}` with a note to run `query` for
the full set. Without `--focus`, `brief` behaves exactly as today (no ranking,
full lists) — a pure superset, so nothing regresses.

### 4. Entity queries

`query --mode entity --entity "[[Auth service]]"` returns every record whose
searchable fields or wiki-links reference the entity, across all projects in
the routed vault, grouped by project. Alias/target normalization matches
`[[Auth service]]` and `[[Auth service|auth]]`; unresolved or ambiguous
entities are reported explicitly, never silently missed. Obeys the existing
`--consent` filter and vault scoping.

### 5. Reindex command

`reindex [--vault-name X]` forces a rebuild and reports record count, index
size, and revision key — for diagnostics and cron warming. Normal reads
rebuild transparently, so this is never required.

## Performance target (honest boundary)

Local focused-brief query p95 under one second on a fixture vault of a few
thousand records, measured after any sync, separately from index build time
and from git network operations. A stale index can never surface a retired
decision or hide an active dispute — the revision key plus direct-scan
fallback guarantee this, and revision-key tests cover checkout, fast-forward,
and content-edit cases.

## Out of scope

Embeddings / vector search (revisit only if keyword ranking demonstrably
fails); cross-vault focused briefs (entity queries stay within the routed
vault per the scope policy); LLM-based ranking or summarization.

## Testing

- Index: build/load, revision-key invalidation on HEAD change, on content
  edit, and on a git checkout; atomic rebuild; direct-scan fallback when no
  revision; `0700` permissions.
- Ranking: golden-fixture ordering for a focus query; safety classes always
  present; stable tie-breaks; omission counts correct.
- Focused brief: superset behavior (no `--focus` == today's brief); caps and
  omission; withdrawn/superseded records excluded.
- Entity query: cross-project match, alias normalization, consent filter,
  ambiguous/unresolved reporting.
- Full regression: existing suites pass unmodified.
