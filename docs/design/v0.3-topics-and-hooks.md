# v0.3 — Topics Across Repos & Capture Hooks

## Status

Shipped in v0.3.

## Problem

v0.2 projects mapped one-to-one with "a codebase area", but real work is
organized by *initiative* ("Payments revamp") spanning several repos, with
several initiatives sharing the same repos concurrently. v0.2's
one-workspace→one-project routing could not express that, and records didn't
say which repo they concerned. Separately, memory coverage depended entirely
on someone remembering to run the brief and capture the session.

## Part 1 — the topic layer

1. **Topic = project.** A project note *is* an initiative, with
   `workspace_repos` listing every repo it spans. No schema migration;
   existing single-repo projects keep working as single-topic projects.
2. **Repo is a stamped facet on records, not a folder.** Record commands
   accept `--workspace <dir>` (agents pass `$PWD`); the CLI derives the
   workspace's normalized `origin` remote and stamps `repos: [...]` on the
   record. Explicit `--repo` flags (repeatable) override or extend. A record
   touching several repos carries all of them — a facet, never a filing
   dilemma. Record bodies append a `Repos: [[<short-name>]]` line so each
   repo becomes an Obsidian graph node.
3. **Brief disambiguation.** Multiple topics legitimately share a repo, so
   workspace routing may become ambiguous. `brief`/`query` accept
   `--project <id>` (workspace then optional); the ambiguity error lists
   candidates so an agent can ask the user and retry.
4. **Topic retirement.** Project notes carry `status: active|done`
   (`--status` on the `project` command). `done` topics are skipped by
   routing and lookup; their records remain readable, and writes to them are
   refused with a revive hint.
5. **Brief grouping.** Briefs include a compact `by_repo` index
   (`{repo: {facts, decisions, sessions}}`) so agents can present activity
   per repository.
6. **Onboarding.** `init-team --identity <name>` makes joining a team vault a
   single command (a personal vault becomes optional), and `init-team`
   scaffolds an `ONBOARDING.md` into the team vault repo so onboarding is
   "here's the repo link".

## Part 2 — capture hooks (Claude Code)

- **Tier 1 — automate reads (SessionStart):** a hook runs
  `brief --workspace <cwd>` and injects the result as additional context, so
  every session starts pre-briefed. Zero risk: reads only; silent when the
  workspace maps to no project.
- **Tier 2 — automate the prompting, keep approval (SessionEnd +
  SessionStart):** SessionEnd leaves a *marker* (not a draft) for
  substantive sessions; the next SessionStart surfaces pending markers with
  an instruction to propose a session recap for the user's approval, then
  delete the marker. The consent gate is untouched — Tier 2 automates
  *remembering to ask*, nothing else.

### Marker store contract

Markers live in `~/.config/context-vault/pending-markers/`, directory
`0700`. A marker is a small JSON file named `<session_id>.json` (exclusive
create = dedup across repeated SessionEnd events) containing only
*references*: session id, transcript path, cwd, ended-at timestamp, schema
version — never transcript content. Cleanup runs at hook invocation time
(markers older than 14 days deleted). Best-effort by design: crashes and
unsupported hosts skip capture; the system never claims complete coverage.

### Substance threshold

SessionEnd writes a marker only when the transcript suggests real work: at
least 5 user messages or 20 KB of transcript.

### Host scope

Claude Code only for the hook surface (`scripts/hooks/`, stdlib Python, JSON
on stdin). Codex has no equivalent hook events; its skill protocol covers
the same moments.

## Compatibility

Existing single-repo projects and all v0.2 behavior unchanged; the whole
prior test suite passes unmodified.
