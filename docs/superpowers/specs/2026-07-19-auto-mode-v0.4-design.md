# Auto Mode (v0.4) — Design

## Status

Revised 2026-07-19 after the consolidated review
(`2026-07-19-auto-mode-v0.4-review.md`), which upheld every maintainer
decision in the log below and required four missing core contracts. All four
(version precedence, SessionEnd execution model, withdrawal vs. incident
remediation, checkpoint ledger) plus the P1 features are now specified.
**One maintainer decision was materially revised by the review** — Q6's
`git revert`-based retract is replaced by `withdraw` (append-only tombstone)
plus a gated `retract --hard`; see the amendment note in the Decision Log.
v0.4 ships **experimental and default-off**.

## Version precedence (supersession matrix)

v0.4 explicitly supersedes the following clauses of its parent specs, for
users who opt into auto mode only (manual mode remains governed by the
originals):

| Parent spec | Clause superseded | v0.4 replacement |
|---|---|---|
| V3 design — V3C | "team vaults are excluded from automatic writes in v3 (personal vault only)" | Auto writes allowed to team vaults per member opt-in, stamped `consent: auto` |
| V3 design — V3C | "facts and decisions remain per-item human approvals permanently" | All record types auto under standing consent (maintainer Q1: uniformity) |
| V3 design — V3C | Headless drafting + pending queue as the primary ambient path | Agent-in-context recording is primary; drafting/markers remain the manual-mode and fallback path |
| V2 design — Schema | "`supersedes` remains the only correction mechanism" | Correction mechanisms are `supersedes` **and** `withdraw` (append-only tombstone); tree-level removal exists only as gated `retract --hard` + the security-incident playbook |
| V2 design — Write protocol | propose → show → approve → record for every write | Applies to manual mode; auto mode substitutes standing consent + visibility + withdrawal |

The parent documents carry pointer notes to this matrix. Invariants **not**
superseded anywhere: Markdown canonicality, append-only history, no-loss
conflicts, client-asserted attribution, dispute surfacing, the two-team-vault
cap.

## Problem

v0.3's write path requires a human "yes" per record. Auto Mode removes the
per-record gate for users who opt in: the agent records at every meaningful
moment; correction happens after the fact. This is a deliberate consent-model
change — from *approval-before-write* to *standing consent + owner
responsibility + cheap correction* — whose risks were examined and accepted
explicitly (Decision Log, Accepted Risks).

## Design summary

- **Per-vault mode.** Each vault entry in config carries
  `"mode": "manual" | "auto"` (default manual). A user can run auto for Team
  A, manual for Team B, auto for personal — independently. A top-level
  default applies to vaults without an explicit mode.
- **Safe activation / kill switch.** Auto is enabled only by a durable,
  explicit command: `auto enable [--vault-name X]` (writes config, prints
  what it means, stamps the consent policy origin). Environment overrides are
  **downgrade-only**: `CONTEXT_VAULT_MANUAL=1` forces manual for a session;
  no environment variable can promote a manual user to auto. `auto disable`
  is immediate.
- **In auto mode the agent records without pausing for approval** — sessions,
  facts, and decisions, to any vault whose mode is auto — and still shows
  what it recorded (visibility without a gate).
- **Provenance is reader-visible, not just stored.** Every auto record is
  stamped `consent: "auto"`, `trigger: <kind>`, `session: <id>`, and (when
  applicable) `source_commit: <sha>`. Briefs display the consent and trigger
  on every row; `query --consent auto|manual` filters; disputes show consent
  per side.
- **Evidence-grade schema for auto facts/decisions.** Beyond free-text
  evidence, auto records carry `basis: observed | inferred | user-stated`
  plus the source artifact (file path, commit SHA, test name) that grounds
  the claim — preserving the evidence-backed-memory premise when no human
  curated the evidence line.
- **Checkpoint chain.** Mid-session records supersede their predecessor
  (sessions gain `supersedes`); one working session collapses to one visible
  record with full history underneath. Records remain new-files-only.
- **Trigger taxonomy** (milestone-driven; no timers, no numeric caps):
  artifact milestones, decision moments, code-commit coupling, PreCompact,
  wrap-up. Full trigger contracts below.
- **Correction:** `withdraw` (append-only tombstone) for wrong records;
  `retract --hard` (tree removal) only within a grace window and only when
  the write receipt proves a record-only commit; a security-incident playbook
  for leaked credentials. Contracts below.
- **Checkpoint ledger** (local, durable) makes all triggers idempotent and
  keeps supersession chains unforked. Contract below.
- **Fire-and-forget with observability.** No ratify chore. `auto status` and
  a non-blocking one-line digest in the next brief report records made,
  unpushed writes, skipped/duplicate triggers, and the last hook failure.
- **Disputes surface, never hold.** Contradictions record and trip the
  normal dispute warning.

## SessionEnd execution model and host capability matrix (review P0-2)

The prior draft claimed SessionEnd "records the recap directly" while also
claiming no headless summarization — an impossibility, since no agent exists
after a session to compose semantic content. The resolved model:

**The final recap is recorded by the live agent, before the session ends.**
The skill defines a *wrap-up trigger*: when work concludes (user signals
done, or the agent finishes its last task), the agent records the final
recap superseding the last checkpoint. SessionEnd's job is **detection, not
composition**: the hook consults the ledger; if the session produced
checkpoints but no final recap, it leaves a *marker* (v0.3 behavior) so the
next session proposes the wrap-up. A hook never invents semantic content.

| Capability | Claude Code | Codex |
|---|---|---|
| Brief injection at start | SessionStart hook | Skill protocol (task-start instruction) |
| Milestone/decision/wrap-up recording | Live agent per skill | Live agent per skill (identical) |
| Code-commit nudge | PostToolUse hook on `git commit` | Skill instruction only (no hook surface) |
| PreCompact checkpoint | PreCompact hook | Not available — no equivalent event; mitigated by milestone frequency |
| Missed-wrap-up detection | SessionEnd hook → marker | None — sessions that end without wrap-up are caught by the next session's brief showing the dangling checkpoint |
| Unsupported host | Skill protocol only; all hook-dependent triggers absent; auto mode still works via agent-side triggers | — |

Degradation rules: if the host cannot give the agent a final tool turn
(crash, kill), the last checkpoint stands as the session's record and the
dangling state is visible in `auto status`. A wrap-up written locally but not
pushed is **pending-sync** (ledger state), never re-marked or duplicated;
the next sync or session delivers it.

## Correction contracts (review P0-3)

**`withdraw <record>` — the standard correction.** Appends a tombstone
record (`type: withdrawal`, `withdraws: <id>`, reason, author, consent) —
a new file, pushed like any record. Withdrawn records: excluded from
current-state briefs and dispute detection; **still present in historical/
temporal queries** (labeled withdrawn-as-of), preserving the bitemporal
model; visible in history forever. This is the only correction path the
skill teaches for ordinary mistakes.

**`retract --hard <record>` — narrow, gated tree removal.** Allowed only
when (a) the ledger's write receipt proves the introducing commit touched
that record file alone, and (b) the grace window (default 10 minutes,
config) has not passed. Reverts and pushes with the `retract:` message
convention the CI validator exempts. Refused otherwise, with `withdraw`
suggested. Honest limits stated in output: history and already-pulled clones
retain the content.

**Security-incident playbook (documentation, not a command).** For leaked
credentials: (1) rotate the credential immediately — treat vault presence as
compromise; (2) `withdraw` the record so current state is clean; (3) only if
warranted, coordinated git-history remediation across clones (filter-repo +
team re-clone), acknowledged as disruptive and rare. No command pretends to
un-leak.

**Record-only commit receipts.** Every auto write commits its record file
alone; the ledger stores record path → commit SHA. This is what makes
`retract --hard` provably safe and retraction auditability possible.

## Checkpoint ledger (review P0-4)

A local, durable, append-only JSONL ledger at
`~/.config/context-vault/ledger/<session_id>.jsonl` (directory `0700`).
Each entry: `session_id`, `trigger` (milestone | decision | git-commit |
precompact | wrapup), `workspace` and routed repo, `source_commit` (for
git-commit triggers), `parent_checkpoint` (record id), `idempotency_key`
(hash of session + trigger + source), `record_path`, `write_commit` (the
receipt), `push_state` (pushed | pending-sync).

Rules: an idempotency key already in the ledger is a skipped duplicate
(counted for `auto status`); a checkpoint may supersede **only** the latest
checkpoint of its own session (validated against the ledger — cross-session
supersession of session records is refused); ledger entries are written
before push and updated after, so retries resume rather than duplicate;
ledgers expire with the marker-store retention policy.

## Trigger contracts (review details)

- **Git-commit trigger** fires only after a *successful* `git commit` in a
  **routed code repository** — never the vault repo, never `revert`/`merge`
  commits, never failed commands, and never a commit SHA already in the
  ledger.
- **PreCompact** records a checkpoint from live context; if the host cannot
  grant a tool turn, it degrades to nothing (the previous checkpoint
  stands).
- **Wrap-up** is an agent obligation in the skill, not a hook product.
- **Substance:** trivial sessions (no milestones, no decisions) produce no
  records; the taxonomy is the volume control (maintainer Q7 — no numeric
  caps).

## Decision Log (the Q&A, preserved for review)

### Q0 — "If multiple memorizations happen per session, is it an append, not a new doc?"

**Answer adopted:** structurally each record is a new file (append-only =
the vault only gains files). The experience of one growing doc is delivered
by the supersession chain; briefs show one visible record per session while
history keeps every checkpoint.

### Q1 — Which record types go auto?

Options: (a) sessions only · (b) sessions + facts · (c) everything.
Assistant recommended (b), flagging auto-recorded decisions as the highest-
risk artifact. **Maintainer: (c) everything — uniformity between manual and
auto.**

### Q2 — Which vaults?

Assistant had argued (and V3 recorded) team-vault exclusion on blast-radius
grounds; maintainer's counterargument: consumption is scoped to devs on the
same/follow-up work, and supersession corrects. Assistant's counters (peak
consumption in the error window; supersession fixes the graph, not work
built on it) acknowledged. **Maintainer: personal auto, and same behavior
for team vaults.**

### Q3 — Who consents for a shared vault?

Assistant proposed a vault-committed policy file. **Maintainer: per-member
choice; no vault-level gate.** `consent: auto` stamp retained for
transparency. (Review round: refined to per-vault member opt-in — the same
person can be auto on one team vault and manual on another.)

### Q4 — Fire-and-forget or ratify chore?

**Maintainer: fire-and-forget; the vault user is responsible and will
revisit for review and edit.** (Review round: augmented with non-blocking
observability — `auto status` + digest — which is information, not a gate.)

### Q5 — Dispute exception?

Assistant recommended holding contradictions. **Maintainer: no — complete
auto, no review at any point.** Contradictions record and surface via the
dispute warning.

### Q6 — Undo ergonomics?

**Maintainer's original decision: a `retract` command that pulls the
record's commit out of the hosted repo (git revert + push) with review in
the command.** **Amended by the consolidated review (P0-3), pending
maintainer confirmation:** plain `git revert` breaks bitemporal queries,
can drag unrelated changes, and cannot un-leak. Replaced by `withdraw`
(tombstone, default) + `retract --hard` (grace-windowed, receipt-gated tree
removal — preserving the original intent where it is provably safe) + the
security-incident playbook.

### Q7 — Volume guardrails?

Assistant proposed a numeric cap. **Maintainer: organize by essential
moments — schema generated → record lands; key feature parts built →
progressive records; session end → final.** Became the milestone taxonomy;
no numeric cap.

### Q8 — Mid-session checkpoints?

**Maintainer: yes; asked for best approaches.** Adopted: milestones +
code-commit coupling + PreCompact + wrap-up, supersession-chained, with
correction commands as the valve. Periodic timers rejected as noise.

## Accepted risks (explicit)

Laid out in full before the decisions and **knowingly accepted** for auto
mode, with listed mitigations only:

1. Wrong claims reach teammates before any human sees them (stamps,
   disputes, supersession, withdraw).
2. Cascading corruption between agents (dispute surfacing, stamps,
   withdraw).
3. Attribution weakening — `author` on records the author never saw
   (`consent: auto` separates vouched from recorded).
4. Leakage — auto content pushes without review; pattern scanning cannot
   catch unknown shapes (patterns still run; withdraw + incident playbook;
   history retains content).
5. Dispute noise if confused agents contradict often (warnings only).
6. No team-level veto — vaults mix gated and auto records per member
   (stamps + brief provenance + CI visibility).

No document may describe auto mode as reviewed or human-gated; the honest
description is "standing consent with owner responsibility, visible
provenance, and append-only correction."

## Mechanics

- **Config:** per-vault `mode` + top-level default; `auto enable/disable`
  writes it durably; `CONTEXT_VAULT_MANUAL=1` downgrade-only override.
- **CLI:** `consent`/`trigger`/`session`/`source_commit`/`basis` stamped by
  the CLI in auto mode (never hand-typed); sessions gain `--supersedes`;
  new commands `withdraw`, `retract --hard`, `auto enable|disable|status`;
  `query --consent` filter; brief rows carry consent + trigger; digest line
  in brief output.
- **Skill:** auto-mode section — trigger taxonomy, no-pause protocol
  (record, show, continue), wrap-up obligation, withdraw protocol, the
  incident playbook pointer, per-host capability notes.
- **Hooks:** PostToolUse (`git commit` filter) nudge; PreCompact checkpoint
  instruction; SessionEnd becomes ledger-checking detector (marker on
  missing wrap-up; pending-sync awareness). All hook additions are
  instructions to the live agent, never content producers.
- **Validator/CI:** `retract:` commit exemption; `withdrawal` record type;
  `consent`/`basis`/`trigger` as allowed fields; author stamps still
  required.
- **Parent docs:** V3 and V2 carry pointer notes to the supersession matrix.

## Testing

- Mode resolution (per-vault, default, downgrade-only env); `auto enable`
  durability; manual mode byte-identical to v0.3 (full regression).
- Stamps present in auto, absent in manual; brief rows show consent/trigger;
  `query --consent` filters.
- Supersession chain + ledger: duplicate idempotency keys skipped;
  cross-session supersession refused; pending-sync resume without
  duplication.
- `withdraw`: excluded from current briefs/disputes, present in temporal
  queries labeled withdrawn; `retract --hard`: allowed only inside grace
  window with record-only receipt, refused otherwise; CI exemption works.
- Git-commit trigger filtering (vault repo, revert, merge, repeat SHA all
  ignored).
- Two-clone integration: auto checkpoint from A visible in B; withdraw from
  A hides it in B's brief on next sync.

## Out of scope

- Vault-level policy gates; ratify chores; dispute holds; numeric caps;
  periodic timers (all explicitly rejected by the maintainer).
- Headless transcript summarization (manual-mode/fallback path only, per
  V3C).
- History rewriting as a product feature (incident playbook only).
