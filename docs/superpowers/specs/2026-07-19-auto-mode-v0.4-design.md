# Auto Mode (v0.4) — Design

## Status

Draft, 2026-07-19, awaiting the maintainer's consolidated review. All
decisions below were made explicitly by the maintainer in design discussion;
the full Q&A (options presented, arguments exchanged, calls made) is preserved
in the Decision Log so the review can audit the reasoning, not just the
outcome.

## Problem

v0.3's write path requires a human "yes" per record. The proposal-riding
pattern (record at the decision moment, one combined yes) reduces friction to
near zero when a human is in the loop — but memory coverage still depends on
those moments happening and being taken. Auto Mode removes the per-record
gate entirely for users who opt in: the agent records at every meaningful
moment, and correction happens after the fact instead of approval before it.

This is a deliberate consent-model change: from *approval-before-write* to
*standing consent + owner responsibility + cheap retraction*. The risks of
that trade were examined explicitly (see Decision Log and Accepted Risks) and
accepted by the maintainer.

## Design summary

- A `mode` setting (`"manual"` default, `"auto"`) in config — per-user, with
  a per-session override (`CONTEXT_VAULT_AUTO=1`). Manual mode is exactly
  v0.3 behavior; nothing changes for users who don't opt in.
- In auto mode the agent records **without pausing for approval** — sessions,
  facts, and decisions alike (uniformity: everything shared in manual is
  shared in auto), to personal and team vaults alike. It still *shows* what
  it recorded (visibility without a gate).
- Every auto-written record is stamped `consent: "auto"` in frontmatter, so
  any reader can forever distinguish "a human vouched for this" from "an
  agent recorded this under standing consent."
- **Checkpoint chain:** mid-session records supersede their predecessor
  (sessions gain the `supersedes` field facts/decisions already have), so one
  working session collapses to one visible record in briefs, with full
  checkpoint history preserved as immutable files underneath. Records remain
  new-files-only; nothing is ever edited in place.
- **Trigger taxonomy** (milestone-driven, no timers, no numeric caps):
  1. **Artifact milestones** — something became true: schema generated,
     migration written, endpoint working, tests green on a feature part,
     branch pushed, PR opened, blocker hit.
  2. **Decision moments** — a design or approach is settled.
  3. **Code-commit coupling** — a `PostToolUse` hook on `git commit` in the
     code repo nudges a vault checkpoint: the user's own "this state is worth
     keeping" judgment is inherited by memory for free.
  4. **PreCompact checkpoint** — a `PreCompact` hook records state right
     before the host compresses its context: the vault captures
     full-fidelity state at exactly the moment agent memory is about to
     weaken.
  5. **Session end** — the final recap supersedes the last checkpoint
     (SessionEnd hook records directly in auto mode instead of leaving a
     marker).
- **`retract <record>`** — the safety valve for eyes-free writes: shows the
  record, confirms, locates the commit that added it, `git revert`s it, and
  pushes. Retract commits use a `retract:` message convention that the CI
  append-only validator recognizes and exempts (tree-level removal;
  git history still preserves everything).
- **Disputes surface, never hold.** An auto-record contradicting an existing
  exclusive fact records anyway and trips the normal dispute warning in
  briefs. No review queue, no held records, anywhere.
- **Per-member consent for team vaults.** Each teammate chooses manual or
  auto for their own writes; there is no vault-level policy gate. The
  `consent: auto` stamp provides transparency to teammates in place of a
  collective gate.
- **Fire-and-forget ownership.** No ratify chore, no post-hoc review ritual.
  The vault owner is responsible for revisiting, superseding, or retracting
  their own records.

## Decision Log (the Q&A, preserved for review)

### Q0 — "If multiple memorizations happen per session, is it an append, not a new doc?"

**Answer adopted:** structurally each record is a new file (append-only =
the vault only gains files; immutability is what keeps merges conflict-free
and history honest). The *experience* of one growing doc is delivered by the
supersession chain: each checkpoint supersedes the previous, so briefs show
one visible record per session while history keeps every checkpoint.

### Q1 — Which record types go auto?

Options: (a) sessions only · (b) sessions + facts · (c) everything.
Assistant recommended (b), flagging auto-recorded *decisions* as the highest-
risk artifact (a "call" nobody made). **Maintainer decision: (c) everything —
sessions, facts, decisions — for uniformity: whatever is shared and memorized
in manual mode is shared and memorized in auto mode.**

### Q2 — Which vaults?

Assistant had earlier argued (and v3's design recorded) that team vaults
should be excluded from automatic writes on blast-radius grounds; the
counterargument raised by the maintainer: consumption is scoped — a wrong
record only reaches devs on the same or follow-up feature, and supersession
corrects it. Assistant's counter-points (peak consumption happens in the
error window; supersession fixes the graph, not work built on it; the
affected devs are exactly the system's audience) were acknowledged.
**Maintainer decision: personal vaults auto (weak-memory risk is minimal),
and the same behavior for team vaults — no personal/team asymmetry.**

### Q3 — Who consents for a shared vault?

Assistant proposed a vault-committed policy file (team decides collectively;
CI flags violations). **Maintainer decision: per-member choice — each
teammate decides auto or manual for themselves; no vault-level gate.** The
`consent: auto` stamp is retained so records remain distinguishable.

### Q4 — Fire-and-forget or write-now-review-later (ratify chore)?

**Maintainer decision: fire-and-forget. The vault user is responsible and
will revisit for any review and edit.** No ratify chore.

### Q5 — Dispute exception (hold contradictions for review)?

Assistant recommended holding contradictions as the one exception.
**Maintainer decision: no — auto is complete auto, no review at any point.**
Contradictions record and surface through the existing dispute warning.

### Q6 — Undo ergonomics?

**Maintainer decision: provide a `retract` command that pulls the record's
commit out of the hosted repo (git revert + push) with a review/confirm step
in the command itself.** Design consequence handled: the CI append-only
check exempts recognized `retract:` commits.

### Q7 — Volume guardrails?

Assistant proposed a numeric per-session cap. **Maintainer decision: organize
by *essential moments* instead — e.g., a database-schema commit should land a
session record as soon as the schema is generated, then progressively as key
feature parts are built, then finally at session end.** This became the
milestone trigger taxonomy; no numeric cap.

### Q8 — Mid-session checkpoints?

**Maintainer decision: yes — wanted; asked for best approaches.** Assistant
presented four: milestone-triggered (recommended core), code-commit coupling
(PostToolUse hook on `git commit`), PreCompact checkpoint (capture state
before host context compression — directly serving the weak-memory concern),
and periodic timers (rejected: records mid-thought states uncorrelated with
meaning). Adopted: milestones + code-commit coupling + PreCompact +
session-end, chained by supersession, with `retract` as the valve.

## Accepted risks (explicit, from the risk discussion)

The following risks were laid out in full before the decisions above and are
**knowingly accepted** in auto mode, with the listed mitigations only:

1. **Wrong claims propagate to teammates before any human sees them.**
   Mitigations: `consent: auto` stamp, dispute warnings, supersession,
   `retract`. Accepted on the maintainer's blast-radius reasoning (impact
   scoped to devs on the same/follow-up work).
2. **Cascading corruption** (agents building on each other's unreviewed
   records). Mitigations: dispute surfacing, stamps, retract. Accepted.
3. **Attribution weakening** — `author` stamps on records the author never
   saw. Mitigation: the `consent: auto` stamp separates "vouched" from
   "recorded"; readers judge accordingly. Accepted.
4. **Leakage** — auto-recorded content is pushed without human review; the
   secret patterns are a narrow net and cannot catch unknown shapes.
   Mitigations: pattern check still runs on every write; `retract` exists;
   git history retains retracted content (tree-level removal only). Accepted.
5. **Dispute noise / cry-wolf** if confused agents contradict often.
   Mitigation: none beyond the warnings themselves. Accepted.
6. **No team-level veto** — a vault's records may mix human-gated and auto
   writes at each member's discretion. Mitigation: stamps + CI visibility.
   Accepted.

No document may describe auto mode as reviewed or human-gated; the honest
description is "standing consent with owner responsibility and retraction."

## Mechanics

- **Config:** `"mode": "auto" | "manual"` (default manual) at top level of
  `config.json`; `CONTEXT_VAULT_AUTO=1` overrides per session. Manual users
  see zero behavior change.
- **CLI:** record commands gain nothing — the agent simply calls them; the
  `consent` stamp is applied by the CLI when mode is auto (config- or
  env-derived), so it cannot be forgotten. `retract <record-path-or-id>`
  added: resolves the commit via `git log --diff-filter=A -- <path>`,
  displays the record, confirms interactively, reverts, pushes, reports.
  Sessions gain `--supersedes` (chain support) — `propose-session`/
  `record-session` accept and validate it like facts/decisions do.
- **Skill:** an "Auto mode" section defines the trigger taxonomy and the
  no-pause recording protocol (record, show what was recorded, continue), and
  the retraction protocol (user says "retract that" → run the command).
- **Hooks:** SessionEnd in auto mode records the final recap directly
  (identity/routing resolved at hook time; falls back to a marker if
  recording fails). New `PostToolUse` hook (matcher `Bash`, filter
  `git commit`) nudges a checkpoint; new `PreCompact` hook instructs a
  checkpoint before compaction. Both are instructions to the agent
  (additionalContext), not headless writers — the agent in context does the
  recording, which is why no transcript summarization is needed.
- **Validator/CI:** recognizes `retract:` commits as exempt from the
  append-only violation check; continues to require author stamps; treats
  `consent` as an allowed record field.
- **Relationship to v3:** this reshapes V3C — the agent-in-context path
  becomes the primary ambient mechanism (no headless drafting needed for it);
  the pending-draft queue remains relevant only for manual-mode users and for
  sessions that end without a recording agent. The v3 design doc gets a
  pointer to this spec after the consolidated review.

## Testing

- Mode parsing and env override; manual mode byte-identical behavior to v0.3
  (full regression unmodified).
- `consent: auto` stamped in auto mode, absent in manual.
- Session supersession chain: checkpoint 2 supersedes 1; brief shows only the
  latest; history files intact.
- `retract`: removes the record from the tree, pushes a `retract:` commit,
  survives the CI append-only check; refuses on unpushed/dirty edge cases
  gracefully.
- Hook additions: PostToolUse nudge fires only on `git commit` commands;
  PreCompact emits checkpoint instruction; SessionEnd records directly in
  auto and falls back to a marker on failure.
- Two-clone integration: auto-recorded checkpoint from clone A visible in
  clone B's brief; retract from A removes it from B on next sync.

## Out of scope

- Vault-level policy gates (explicitly rejected — per-member consent).
- Ratify/review chores for auto records (explicitly rejected).
- Dispute holds (explicitly rejected).
- Numeric volume caps (replaced by milestone triggers).
- Periodic timer checkpoints (rejected as noise).
- Headless transcript summarization (superseded by agent-in-context
  recording for auto mode; remains a V3C fallback for manual mode).
